Always clean up data generated by benchmark/pipeline runs once it's no longer
needed:
- App output files (e.g. h5bench `.h5` files) — delete after the run's trace
  has been captured; the data file itself is rarely needed once traced.
- Raw traces — once split/compacted AND the compact version is verified to
  work, the raw traces can be deleted to save space. **Always ask before
  removing raw traces** — never delete silently, even though this is a
  standing default policy.
- Test/calibration files (single-process and 1-node validation runs before
  scaling to full runs) — clean up immediately after each test, don't let
  them accumulate.

**Why:** discovered 2026-07-10 during h5bench session — misconfigured stuck
jobs silently consumed 24TB on `/p/lustre5` (pushing the whole filesystem
OVER its 100TB quota, 9.9 days to remove 1.4TB before hard enforcement) and
75GB in the session workspace, before the mistake was caught. Filesystem
quotas are shared/site-wide resources, not unlimited.

**How to apply:**
1. Before any run that will write significant data, check quota headroom
   first: `quota -v` (shows all mounted filesystems: home `/g/g92`,
   `/p/lustre5`, workspace mounts, etc.) — and size the run's config to fit
   comfortably within remaining headroom, not just against an assumed budget.
2. If more data/headroom is genuinely needed than is available, do not just
   proceed anyway — **check current quota usage and figure it out together
   with the user** (ask, don't assume you can exceed it or that someone else
   will clean up later).
3. Clean up immediately after each test/run once its purpose is served
   (trace captured, compact verified, results extracted) — don't let
   intermediate artifacts accumulate across a long multi-stage session.
4. Note: the Bash tool's destructive-action permission classifier may block
   `rm -rf` even on in-project-root workspace paths and even after the user
   has confirmed via a question — if blocked, ask the user to run the
   cleanup command themselves rather than trying to work around the block.

See also [[feedback_lustre_io]] (Lustre is for app data, not traces) and
[[feedback_privacy_anonymous]] (workspace is gitignored/session-scoped).
